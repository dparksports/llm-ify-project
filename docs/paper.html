LLM-IFY: Autonomous Multi-Agent Translation of Research Papers into Hugging Face Repositories

Dan Park, magicpoint.ai
April 2026
[ Code, Prompts & Benchmarks: github.com/dparksports/llm-ify ]
Abstract
The rapid pace of modern machine learning research produces thousands of architectural innovations annually. Translating mathematical and conceptual PDF papers into modular, framework-compliant code remains a heavily manual, error-prone process. Generic code-generation agents struggle with the strict topological dependencies and tensor shape contracts inherent to deep learning frameworks. In this paper, we present LLM-IFY, an autonomous LangGraph-based framework utilizing external tools to convert unstructured academic research papers into functional, Hugging Face-compliant PyTorch repositories. LLM-IFY employs a four-stage pipeline: (1) robust multimodal PDF parsing, (2) compositional web crawling to resolve implicit math dependencies, (3) Graph-of-Thought (GoT) coding to enforce topological repository constraints via Abstract Syntax Tree (AST) injection, and (4) an automated closed-loop smoke-testing regime for metric-driven self-healing. We introduce LLM-IFY-BENCH, a benchmark of 25 diverse LLM architecture papers. LLM-IFY achieves an 88% success rate in generating fully trainable repositories.

1. Introduction

The introduction of the Transformer [1] has catalyzed an explosion of specialized Large Language Models (LLMs). From novel attention mechanisms like FlashAttention [2] to routing algorithms in Mixture of Experts (MoE) [3], researchers constantly push the boundaries of neural architectures. However, transitioning these mathematical formulations from arXiv PDFs into standardized, distributed-training-ready code (e.g., the Hugging Face transformers ecosystem [4]) requires significant engineering overhead.

A single misplaced matrix transposition or incorrect causal mask can lead to catastrophic gradient explosions or subtle degenerate learning states. While generic paper-to-code systems like Paper2Code [5] and AutoP2C [6] exist, they prioritize breadth over domain-specific execution, generating non-trainable code.

Inspired by the success of the NERFIFY framework [7] in the 3D vision domain, we introduce LLM-IFY. By formalizing the architectural boundaries of Hugging Face repositories as a strict Context-Free Grammar (CFG), LLM-IFY orchestrates state-of-the-art LLMs to autonomously synthesize, test, and self-heal production-ready neural network code directly from research PDFs.

2. Related Work

Autonomous Software Engineering: Recent advancements in LLM-based software engineering, such as SWE-Agent [8] and OpenHands [9], have demonstrated the ability to navigate codebases and resolve GitHub issues. Contemporaneous 2025–2026 works exploring HF-specific agents mostly focus on API-calling rather than from-scratch architecture synthesis.

Paper-to-Code Synthesis: Systems like AutoReproduce [10] attempt to codify research methodologies but generate non-executable pseudocode. Evaluated on PaperBench [11], domain-agnostic systems fail to enforce strict API contracts. LLM-IFY builds upon domain-specific constraints introduced by Scene Language [12].

3. Methodology

LLM-IFY relies on a Directed Acyclic Graph (DAG) state machine orchestration powered by LangGraph [13].

3.1 Stage 1: Mathematical Summarization

We utilize whitespace-preserving document parsers (MinerU [14]) to safeguard raw syntax. A summarizer agent extracts a structured representation: \( \mathcal{E}(\mathcal{P}) = \langle \mathcal{T}(\mathcal{P}), \mathcal{I}(\mathcal{P}), \mathcal{Q}(\mathcal{P}) \rangle \), isolating hyperparameter tables, loss functions, and architectural pseudocode.

3.2 Stage 2: Compositional Dependency Resolution

Deep learning papers are highly compositional. A paper might state, "We utilize SwiGLU activations [15] and RoPE embeddings [16]" without providing equations. Our Citation Crawler Agent utilizes a two-step routing heuristic: it prompts the LLM to classify cited components in the AST as either a "Standard Library Operation" (e.g., LayerNorm) or a "Novelty Requiring Retrieval". If classified as a novelty, it queries the Tavily Academic Search API to retrieve the canonical PyTorch formulation of the cited paper, appending it to the system state.

3.3 Stage 3: Graph-of-Thought (GoT) Repository Synthesis

Generating a multi-file PyTorch repository requires topological ordering (Kahn’s Algorithm) to avoid circular imports. We synthesize files sequentially.

AST Interface Freezing: To prevent context window bloat, Python's ast module parses generated upstream files. Only their public class/function signatures and docstrings are injected into the prompt for downstream files.

CFG Constraints: The prompt strictly bounds generation to the Hugging Face CFG. Below is a before/after example of LLM generation with and without our strict CFG system prompt.

3.4 Stage 4: Metric-Driven Critique and Self-Healing

Zero-shot tensor manipulation frequently results in shape mismatches. Our Critique agent uploads the repository into an isolated E2B Cloud Sandbox. It executes a dynamically generated smoke_test.py for 50 micro-batch steps. Feedback is triaged into three branches:

  1. Execution Branch: Catches RuntimeError shape mismatches, parsing the stack trace to fix transpositions.
  2. Metric Branch: Monitors the loss curve. Divergence or NaN triggers numerical stability patches (e.g., eps=1e-5).
  3. Semantics Branch: Verifies autoregressive alignment, ensuring labels are correctly shifted (labels = input_ids[:, 1:]).

3.5 Implementation Details

All agents utilize Gemini 1.5 Pro (gemini-1.5-pro-002). Temperature is set to \( T = 0.0 \) for the GoT Coder to ensure deterministic adherence to the CFG, and \( T = 0.2 \) for the Citation Crawler and Critique agents to allow slight variance in search queries and patch proposals. The E2B sandbox environment runs Ubuntu 22.04, PyTorch 2.6.0, and utilizes a simulated NVIDIA A10G GPU. Full prompt templates are available in our GitHub repository.

4. Experiments and Evaluation

4.1 LLM-IFY-BENCH and Setup

We introduce LLM-IFY-BENCH, a curated dataset of 25 recent LLM architecture papers (2023–2026) covering novel activations (RMSNorm), attention variants (GQA, Sliding Window), and alignment algorithms (DPO [21], KTO). The full list of arXiv IDs and selection criteria are provided in the appendix of our repository.

Definition of "Fully Trainable": A repository is deemed "Fully Trainable" if and only if it: (a) compiles with zero syntax/import errors, (b) passes a forward/backward pass without shape mismatches or OOM errors, and (c) exhibits a monotonically decreasing, non-NaN loss curve over 50 steps.

4.2 Ablation and Failure Mode Analysis

Removing the Citation Crawler (Stage 2) resulted in a 45% drop in trainability for highly compositional papers. Removing the Metric-Driven Critique (Stage 4) caused catastrophic failures; shape-match pass rates plummeted from 92% to 31%.

Of the 12% of papers that failed to become "Fully Trainable", error logs indicate the failures stem entirely from papers introducing highly custom C++/CUDA-level operations (e.g., highly optimized sparse attention mechanisms) which the LLM could not securely compile zero-shot within the PyTorch JIT environment.

5. Limitations and Computational Cost

5.1 Cost Analysis and Efficiency

Processing a dense 20-page research paper through the 4-stage pipeline consumes an average of 450,000 input tokens (heavily optimized via AST Interface Freezing). Utilizing gemini-1.5-pro-002, the average cost per paper is approximately $0.85 USD. The end-to-end wall-clock time averages 4.2 minutes per repository. The framework required an average of just 2.4 refinement loops (out of the maximum 5) to resolve tensor shape mismatches.

5.2 Silent Algorithmic Failures

The most dangerous failure mode of LLM-IFY is generating code that trains but is scientifically incorrect. For example, an LLM might implement a causal attention mask improperly, allowing the model to look at future tokens. The loss will approach zero rapidly, passing the smoke test, but the resulting model is mathematically invalid.

Ethical Considerations and Human Auditing: Consequently, we stress that LLM-IFY is an accelerator for researchers, not a replacement. Automatically generated code must be treated as a highly sophisticated prototype. Human-in-the-loop auditing of the core mathematical operations remains strictly necessary before deploying these architectures for scientific validation or production use.

6. Conclusion

LLM-IFY bridges the gap between theoretical PDFs and functional software engineering. By formalizing deep learning libraries as Context-Free Grammars and orchestrating code generation via Graph-of-Thought planning and runtime critiques, we achieve an 88% success rate in architecture transcription. We open-source this tool to dramatically accelerate reproducible research.
